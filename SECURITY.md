# Security Policy

Do not open public issues containing vulnerabilities, credentials, private translation content, or malicious fixtures that create immediate exploitation risk. Use GitHub private vulnerability reporting once the remote is available.

Supported releases and a response SLA will be documented before the first public prerelease. Until then, only the current `main` branch receives security fixes.

Reports should include affected revision, impact, safe reproduction steps, and suggested mitigations. Never include real provider keys or private documents.
